# LocalScript — визуализация архитектуры (нотация C4)


**Репозиторий:** LocalScript — локальный AI-агент для Lua (FastAPI + Ollama, без внешних LLM API в runtime). Контракт API: `localscript-openapi.yaml` в корне репозитория.




## Рисунок 1 — C4 Level 1 (контекст системы)



На диаграмме показан **актор** (пользователь или интеграция) и **программная система LocalScript** как единый чёрный ящик. Система решает задачу локальной генерации и доработки Lua-кода под LowCode-сценарии. В runtime **не используются внешние облачные LLM API**; инференс выполняется только через локально развёрнутый **Ollama**. Обращение к backend возможно через **Streamlit** (браузер), через **CLI** (`demo_cli.py` в терминале) и/или через **прямой HTTP API** (`/generate`, `/refine`, `/debug`, `/health`).

---

## Рисунок 2 — C4 Level 2 (контейнеры)



Граница **«LocalScript»** объединяет три **контейнера**:

| Контейнер | Технологии | Назначение |
|-----------|------------|------------|
| Web UI | Python, Streamlit (`scripts/demo_streamlit.py`) | Чат-интерфейс, история запросов, вызовы REST к backend. Обычно слушает **порт 8501** на хосте и **не** входит в стандартный `docker compose` как сервис. |
| CLI UI | Python (`scripts/demo_cli.py`) | Тот же REST API, что и у Streamlit, но из **терминала** (интерактивный клиент); тоже на хосте, не в compose. |
| App | Python 3.11, FastAPI, Uvicorn | Stateless HTTP API, сборка промптов, пайплайны, цикл repair, `luac`, статические запреты, опциональный линтер, опциональная **семантическая** проверка через песочницу и `wf` из контекста. |
| Ollama | образ `ollama/ollama`, GGUF на GPU | Локальный инференс; в compose доступен приложению по внутреннему адресу, на хост проброшен **11434**. |

Связи: пользователь → Web UI → App; пользователь → CLI UI → App; пользователь может обращаться к App **напрямую** (curl, SDK, eval); App → Ollama (`/api/chat`).

---

## Рисунок 3 — C4 Level 3 (компоненты)



Внутри контейнера **App** выделены компоненты, соответствующие модулям в `localscript-agent/app/`:

| Компонент | Файлы | Роль |
|-----------|-------|------|
| HTTP API | `main.py` | Маршруты, общий `httpx.AsyncClient`, при настройке — прогрев Ollama при старте. |
| Пайплайн | `pipeline.py` | Сценарии generate / refine / debug; `run_repair_loop`; извлечение JSON-**context** из текста промпта для последующих проверок. |
| Клиент Ollama | `ollama_client.py` | Запросы к `/api/chat`, параметры контекста/генерации из настроек. |
| Промпты и разбор | `prompts.py`, `generate_parse.py`, `extract.py` | Сообщения для модели; разбор ответа (код / уточнение); извлечение Lua. |
| Валидация кода | `code_checks.py`, `validate.py`, при необходимости `semantic.py` | Статика, синтаксис `luac`, опциональный линтер; при наличии спецификации в **context** — семантическая проверка с запуском Lua в песочнице (`sandbox.py` вызывается из `semantic.py`). |
| Контракты и настройки | `models_io.py`, `config.py` | Pydantic-схемы и переменные окружения. |

Снаружи показаны **Streamlit Web UI**, **CLI UI** (`demo_cli.py`) и **Ollama** — оба UI-контейнера являются **клиентами** того же HTTP API, что и `curl`/SDK.

## Рисунок 4 — развёртывание (deployment)



На схеме **хост**: внутри **Docker Compose** размещены контейнеры **app** (проброс **8080**) и **ollama** (**11434**, GPU, том данных). Отдельно на хосте показаны **опциональные** процессы **Streamlit** и **CLI-клиент**, оба ходят в App по `http://127.0.0.1:8080`. Так отражено разделение: compose поднимает backend; демо-интерфейсы — с хоста при необходимости.

---

## Рисунок 5 — динамика (сценарий generate)



Упрощённая **динамическая** диаграмма (нумерованные взаимодействия в стиле C4 Dynamic) для основного сценария **POST /generate**: клиент → API → пайплайн → промптинг/парсинг ↔ Ollama → проверки и repair. Поток **идентичен** для Streamlit и CLI: оба только отправляют те же HTTP-запросы к App.

## Сводная таблица публичного HTTP API

| Метод | Путь | Назначение |
|-------|------|------------|
| GET | `/health` | Статус сервиса, доступность Ollama, готовность модели, ключевые параметры инференса |
| POST | `/generate` | Новая задача; возможен ответ `clarification` до выдачи кода |
| POST | `/refine` | Доработка по истории refinement |
| POST | `/debug` | Анализ переданного Lua и предложение исправления |

## Ограничения и допущения (для жюри)

- **Локальность:** внешние LLM API в рабочем контуре не используются; используется только Ollama с локальной моделью (по условию — `qwen2.5-coder:7b` и фиксированные `num_ctx` / `num_predict`).
- **Stateless API:** состояние диалога клиент передаёт в теле запроса (`clarification_history`, `refinement_history`, `debug_history`).
- **Семантика:** опциональные проверки включаются глобально настройками или при передаче структурированного **context** в тексте промпта (в т.ч. через GUI), см. `semantic.py` и `code_checks.run_all_checks`.
- **Оценка на бенчмарке:** скрипт `scripts/eval_public.py` может использовать `sandbox.py` для задач с типом `sandbox` в JSON — это отдельный контур от HTTP, но тот же движок проверок.
