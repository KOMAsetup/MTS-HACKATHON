# Как работать в связке с ассистентом (проверка кода, отладка, метрики)

Цель: чтобы в Cursor можно было самостоятельно гонять тесты, линтер, бенчмарк и итерировать промпт/конфиг.

### Продукт vs «подогнать тесты»

Итоговая система — это **генерация Lua + валидация** (статика, `luac`, sandbox), согласованная с **реальным LowCode** и PDF/платформой, а не максимизация `pass` на восьми строках `expected_contains`.

- **Источник истины по уровню доверия:** корректность в **sandbox** и **синтаксис** выше, чем эвристики по подстрокам; подстроки в `public_tasks.json` — лишь **дешёвый дымовой** сигнал, когда нет эталонного выхода.
- **Промпты и few-shot** должны отражать **настоящие правила платформы** (в т.ч. `lua{...}lua`, `_utils.array`, запреты), а не только то, что проверяет JSON.
- **Менять бенчмарк** имеет смысл только если эвристика **ошибочно противоречит** PDF/API (как с ложным `package.` или литералом даты в исходнике); иначе лучше усиливать **проверку по смыслу** (например, отдельный скрипт, который сравнивает результат в sandbox с ожидаемой строкой ISO), а не раздувать `expected_contains`.

Дообучение (QLoRA и т.д.) имеет смысл на **размеченных вручную** эталонах в `data/synthetic/`, а не на том, чтобы модель заучила подстрочные тесты.

## 1. Что должно быть запущено у тебя

Пока идёт сессия с ассистентом, держи поднятым стек (или нативный Ollama + uvicorn):

```bash
cd /path/to/<корень-репозитория>/localscript-agent
docker compose up --build
```

Проверки:

```bash
curl -s http://127.0.0.1:8080/health
curl -s http://127.0.0.1:11434/api/tags   # не пустой список models
```

Если `models` пустой — модель не скачана:

```bash
sudo docker exec -it localscript-agent-ollama-1 ollama pull qwen2.5-coder:7b
```

(имя контейнера уточни: `sudo docker ps`.)

---

## 2. Что ассистент может делать сам в репозитории

Из каталога `localscript-agent/` (и только его — остальные воркспейсы не трогаем):

| Действие | Команда |
|----------|---------|
| Линт + формат + тесты | `./scripts/quality.sh` |
| Бенчмарк через HTTP (нужен `:8080`) | `python3 scripts/eval_public.py --http --base-url http://127.0.0.1:8080` |
| Быстрый смоук (2 задачи) | `python3 scripts/eval_public.py --http --base-url http://127.0.0.1:8080 --tasks benchmarks/synthetic_smoke.json` |
| In-process eval (Ollama на хосте, не в Docker) | `python3 scripts/eval_public.py` с `OLLAMA_HOST` на твой Ollama |
| Проверка Ollama | `./scripts/check_ollama.sh` |

Метрики из JSON вывода `eval_public` (поле `metrics`): **M1** — `syntax_ok`, **M1b** — `static_ok`, **M2** — `sandbox_ok` (где есть sandbox), **M3** — `heuristic_ok`; сводка по `passed/total`; дополнительно `infra_fail`, `generation_error`, `model_error`, `latency_avg`, `latency_p95`. Детали по задачам — в `results[]` (`latency_s`, `errors`, `pass`).

Фиксация прогонов: `experiments/runs.csv`, выводы — `docs/experiments/SUMMARY.md`, эталон — `docs/experiments/BEST_CONFIG.md`.

---

## 3. Что нужно от тебя в начале «рабочей» сессии

Напиши одним сообщением, например:

- «Compose поднят, модель есть, health ok»  
- или приложи вывод `health` + `api/tags` (как ты уже делал).

Так ассистент не тратит время на выяснение, почему пустой `models` или недоступен порт.

Опционально:

- **`docker` без sudo** — `usermod -aG docker` + перелогин, чтобы команды в сессии не упирались в права.
- **Conda** для прогона `eval_public` с хоста: `conda activate localscript-agent` в том же окружении, где стоят `lua`/`luac`.

---

## 4. Типичный цикл улучшения

1. `./scripts/quality.sh` — не ломаем регресс.
2. Полный или смоук `eval_public.py --http`.
3. Разбор `results[].errors` в JSON.
4. Правки `app/prompts.py`, при необходимости `app/config.py` / `.env` (temperature, repair, `num_batch`).
5. Повтор 1–4; лучший прогон — строка в `runs.csv` + обновление `SUMMARY.md` / `BEST_CONFIG.md`.

---

## 5. Ограничения среды ассистента

- Нет доступа к твоему экрану и долгоживущим процессам, если ты их не запустил; **инференс = твой Docker/Ollama**.
- Долгие `docker pull` / `ollama pull` могут обрываться по таймауту — надёжнее один раз подтянуть модель у себя (см. выше).

Если это соблюдено, можно прямо писать: «прогони quality + полный eval и предложи правки по провалам» — дальше идёт плановая итерация.
